# Integration QA — 경계면 교차 검증

## 핵심 방법론: 경계면 교차 비교

QA의 핵심은 "존재 확인"이 아니라 **경계면 양쪽을 동시에 열어 대조**하는 것이다. 파일이 존재한다 ≠ 동작한다. 각 모듈이 단독으로 옳아도 경계면의 암묵적 가정이 어긋나면 통합이 깨진다.

이 프로젝트의 경계면 지도:

| # | 경계면 | 대조 대상 A | 대조 대상 B | 전형적 버그 |
|---|--------|------------|------------|------------|
| 1 | 수집 ↔ 타입 | `data/*.json` 실제 필드 | `src/types/content.ts` | 옵셔널 필드를 필수로 선언, 문자열 숫자를 number로 선언 |
| 2 | 큐레이션 ↔ 원본 | `curated.json`의 videoId | `videos.json`+`shorts.json`의 videoId | 존재하지 않는 영상 참조 (dangling reference) |
| 3 | 데이터 ↔ 훅 | `lib/content.ts` 반환 shape | 소비 컴포넌트의 구조분해 | 이름이 비슷한 다른 필드 (`viewCount` vs `views`) |
| 4 | 트랙 ↔ 콘텐츠 | tracks의 step videoId | curated items | 트랙이 unclassified 영상 참조 |
| 5 | 플레이어 ↔ 데이터 | embeddable 필드 | 플레이어 폴백 분기 | embeddable=false 처리 누락 |
| 6 | 디자인 ↔ 구현 | 디자인 토큰 정의 | 컴포넌트 스타일 값 | 토큰 우회 하드코딩 |

## 검증 절차 (모듈 단위, 순서 고정)

1. **Shape 대조** — 경계면 지도에서 해당 모듈이 걸친 행을 찾아 양쪽 파일을 열고 필드 단위 대조. 실데이터 최소 5건 + 엣지 케이스(tags 없음, description 빈 값, isShort null) 포함
2. **기계 검증 실행** — `npm run build`, `npx tsc --noEmit`. 출력 전문을 증거로 보존
3. **기능 동선 확인** — 기능 명세의 완료 기준 항목별 실제 확인
4. **엣지 주입** — 빈 배열, 임베드 불가, 초장문 제목(40자+)을 데이터에 주입해 UI 깨짐 확인

## Incremental QA 원칙

전체 완성 후 일괄 검증이 아니라 **모듈 완성 직후 즉시 검증**한다. 늦게 발견된 경계면 버그는 그 위에 쌓인 코드 전체를 재작업시킨다. 모듈 완성 보고 = QA 트리거.

## 결함 보고 표준

`_workspace/04_qa_report_{module}.md`:

```markdown
# QA Report: {module} ({날짜})

## 요약: PASS | FAIL (BLOCKER n건 / MAJOR n건 / MINOR n건)

## 결함 목록
### [BLOCKER] {한 줄 제목}
- 재현: {정확한 경로·명령}
- 기대: / 실제:
- 추정 원인: {파일:라인 또는 "미상"}

## 검증 증거
- 실행한 명령과 출력 요약 (빌드 로그, 대조한 파일 쌍)
- 검증하지 못한 항목과 사유  ← 빈 칸으로 두지 말 것
```

심각도: **BLOCKER**=핵심 동선 차단·빌드 실패·데이터 오염 / **MAJOR**=기능 오동작이지만 우회 가능 / **MINOR**=시각·문구 결함.

## 회귀 검증

재호출 시 이전 보고서의 결함 목록부터 재확인한다. 수정 확인 → `RESOLVED`, 미수정 → `REOPEN` 표기 후 신규 검증으로 진행. 수정이 다른 곳을 깨는 경우가 흔하므로 수정된 파일의 경계면도 재대조한다.

## 거짓 통과 방지

- "아마 되겠지"는 미검증이다 — 검증 증거 섹션에 실행 기록이 없는 항목은 PASS로 표기하지 않는다
- 재현 안 되는 결함은 닫지 않고 `INTERMITTENT`로 유지한다 — "간헐적"은 "조건 미파악"의 다른 말이다
- 검증 환경이 막히면(의존성 등) 막힌 사실 자체를 보고서에 기록한다
